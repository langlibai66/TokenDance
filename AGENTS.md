# Agents 协作说明

本文件用于约束和提示在本项目中工作的 AI Agent / 开发协作者。当前仓库以 `douyin-base` 为前端展示工程，旁边的 `../flow_agent` 为 Python 数据分析工程。若后续业务边界、数据协议或部署方式发生变化，请优先更新本文档。

## 项目结构

- `douyin-base/`：前端展示工程，负责移动端信息流、页面交互和数据可视化展示。
- `../flow_agent/`：Python 工程，负责数据分析、数据处理、视频/内容分析等后端或离线分析任务。
- `public/data/`：前端本地静态数据目录，包含视频、用户、商品等 JSON/Markdown 数据。
- `src/`：前端源码目录。
  - `src/pages/`：页面级视图。
  - `src/pages/home/slide/`：推荐流内容卡片与滑动内容。
  - `src/components/`：通用组件。
  - `src/api/`：前端请求封装。
  - `src/mock/`：本地 mock 数据接入。
  - `src/store/`：Pinia 状态管理。
  - `src/utils/`：工具方法。

## 工程定位

### douyin-base

`douyin-base` 是基于 Vue 3 + Vite 的前端展示项目，主要目标是提供类似抖音/TikTok 的移动端信息流体验。当前项目已包含电动牙刷对比卡片等定制内容，前端需要重点保证：

- 移动端优先的展示效果。
- 推荐流上下滑、卡片内部左右滑等手势体验稳定。
- 数据展示清晰，不把分析逻辑硬编码进 UI。
- 页面表现尽量贴合原有抖音式沉浸体验。

### flow_agent

`flow_agent` 是 Python 工程，用于数据分析、内容分析、数据清洗、指标计算等任务。该工程应尽量产出结构化结果，供 `douyin-base` 读取和展示。

当前已知信息：

- 工程路径：`../flow_agent`
- Python 要求：`>=3.11`
- 包管理/运行建议：`uv`
- 测试框架：`pytest`

## 协作边界

- 前端展示逻辑放在 `douyin-base`。
- 数据分析、数据清洗、模型调用、视频分析等逻辑放在 `flow_agent`。
- 不要在前端组件里直接塞入复杂分析逻辑；前端只消费明确的数据结构。
- 不要让 `flow_agent` 直接依赖前端运行时环境。
- 如需让两个工程联动，优先通过 JSON、CSV、Markdown 或明确的 API/文件协议传递数据。
- 改动跨两个工程时，需要同时说明：
  - 数据从哪里产生。
  - 数据写到哪里。
  - 前端如何读取。
  - 字段含义和兼容性要求。

## 常用命令

### douyin-base

```bash
npm install
npm run dev
npm run build
npm run type-check
npm run lint
npm run format
```

本地开发服务默认使用 Vite，端口配置在 `vite.config.ts` 中，当前为 `3000`。由于项目是移动端展示，调试时应使用浏览器移动设备模式检查主要页面。

### flow_agent

```bash
cd ../flow_agent
uv sync
uv run pytest
```

如果新增 Python 依赖，应写入 `pyproject.toml`，并保持 `uv.lock` 同步。

## 前端开发约定

- 优先复用现有组件、路由、状态管理和工具函数。
- 样式应贴合已有移动端视觉和交互，不随意引入大型 UI 框架。
- 新增展示数据时，优先把静态数据放入 `public/data/`，或通过已有 mock/request 机制接入。
- 修改推荐流内容时，重点检查：
  - 上下滑切换是否正常。
  - 横向滑动是否与页面返回、用户页滑动冲突。
  - 首屏加载、图片加载和空状态是否合理。
  - 移动端尺寸下文本是否溢出。
- TypeScript/Vue 文件应遵循项目现有格式，并在提交前运行必要检查。

## Python 开发约定

- 分析逻辑应拆分为可测试函数，避免把所有逻辑写进脚本入口。
- 输出给前端的数据应稳定、可复用、可追踪来源。
- 对外输出字段建议包含必要的元信息，例如：
  - `id`
  - `title`
  - `source`
  - `metrics`
  - `summary`
  - `updated_at`
- 新增数据格式时，需要同步补充字段说明，避免前端只能靠猜。
- 新增测试放入 `tests/`，并使用 `uv run pytest` 验证。

## 数据协议建议

当前两个工程的连接方式尚未完全固定。默认建议如下：

- `flow_agent` 负责生成结构化数据文件。
- `douyin-base` 负责读取并展示这些数据。
- 数据文件优先使用 JSON；如需人工阅读和维护，可配套 Markdown 说明。
- 前端字段命名尽量保持稳定，避免频繁改动导致展示层反复适配。
- 若字段有枚举值、单位、评分范围或排序规则，应在数据旁边写清楚。

## 质量检查

改动完成后，根据影响范围选择检查项：

- 只改文档：检查 Markdown 内容是否准确、无明显错别字。
- 改前端逻辑：至少运行 `npm run type-check`，必要时运行 `npm run build`。
- 改前端样式/交互：启动 `npm run dev`，使用移动端视口检查关键页面。
- 改 Python 分析逻辑：运行 `uv run pytest`。
- 改数据结构：同时检查 Python 输出和前端读取逻辑。

## Agent 工作要求

- 开始改动前先阅读相关文件，不要只凭猜测修改。
- 保持改动范围小而明确，避免顺手重构无关模块。
- 不要修改 `node_modules/`、`.venv/`、构建产物或锁文件，除非任务明确需要。
- 不要删除用户已有改动；遇到未解释的本地变更时，应保留并绕开。
- 对跨工程改动，要在回复中明确列出影响的工程和文件。
- 如果发现需求信息不足，优先用合理假设推进，并把假设写清楚；影响较大时再询问用户。
